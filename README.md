#  高并发 C++ HTTP 服务器 —— 基于 Reactor 模型

> 一个从零实现的轻量级 HTTP 服务器，核心原理与 Nginx 一致，基于 Linux epoll 事件驱动模型，支持高并发连接处理。
>
>

---

## 这个服务器程序是什么？

这是一个 **完全使用 C++ 从零编写的 HTTP 服务器程序**。

你平时访问网站时，浏览器发出的请求需要一个 "服务器程序" 来接收并返回网页内容。
常见的服务器程序有 **Nginx、Apache** 等，而这个项目做的就是同样的事情：
浏览器输入网址→

发送 HTTP 请求→

【本服务器】监听端口，接收请求→

解析请求，找到对应的 HTML/CSS/JS/图片文件→

返回 HTTP 响应→

浏览器渲染页面→


**简单来说：阉割版 Nginx。**

---

##  它能干什么？

| 能力 | 说明 |
|------|------|
|  **处理 HTTP 请求** | 解析浏览器发来的 HTTP GET 请求 |
|  **返回静态网页** | 返回 HTML、CSS、JS、图片等静态资源 |
|  **支持高并发连接** | 基于 epoll 事件驱动，可同时处理大量客户端 |
|  **线程池处理请求** | 主线程监听事件，工作线程处理业务，互不阻塞 |
|  **自动清理超时连接** | 定时器检测非活跃连接并自动关闭，防止资源泄漏 |
|  **完整日志记录** | 每个请求的时间、IP、URL 都会被记录到日志文件 |
|  **404 错误处理** | 请求不存在的资源时，正确返回 404 状态码 |
|  **真实部署运行** | 已部署在腾讯云服务器上，承载个人主页的真实访问 |

##  核心功能详解

### 1. epoll 事件驱动（ET 模式 + EPOLLONESHOT）

- 使用 **边缘触发（ET）** 模式，事件只通知一次，减少系统调用次数
- 配合 **EPOLLONESHOT**，保证每个连接在同一时刻只被一个线程处理
- 相比 select/poll 的 O(n) 遍历，epoll 只处理活跃连接，效率 **O(1) 注册 + O(k) 返回**（k 为活跃数）

### 2. 线程池并发处理

- 主线程（Reactor）只负责 **事件监听和分发**，不处理业务逻辑
- 工作线程从任务队列中取出请求进行处理
- 使用 **mutex + condition_variable** 实现线程同步
- 避免了为每个请求创建/销毁线程的开销

### 3. HTTP 请求解析与响应

- 解析 HTTP 请求行，提取 **请求方法、URL、协议版本**
- 根据 URL 定位静态文件并读取返回
- 支持 **200 OK / 404 Not Found** 状态码
- 响应头包含 **Content-Type、Content-Length、Connection** 等字段

### 4. 最小堆定时器

- 使用 **最小堆（Min-Heap）** 管理所有连接的超时时间
- 堆顶始终是最早过期的连接，查找最小值 **O(1)**
- 插入和删除操作 **O(log n)**
- 每次事件循环后自动 **tick()**，清理所有过期连接
- 防止恶意连接或僵尸连接占用服务器资源

### 5. 日志系统

- 支持 **终端输出 + 文件写入** 双通道
- 每条日志包含 **精确时间戳 + 日志级别（INFO/ERROR）**
- 使用 **互斥锁** 保证多线程写入安全
- 便于服务运行监控与问题排查

##  压力测试

**测试环境**：腾讯云轻量服务器 4 核 4G / Ubuntu / Intel Xeon Platinum

**测试工具**：wrk

| 并发连接数 | QPS (请求/秒) | 平均延迟 | 最大延迟 | 超时数 |
|:---:|:---:|:---:|:---:|:---:|
| 100 | **5,857** | 25.37ms | 111.91ms | 0 |
| 500 | **5,965** | 49.22ms | 1.74s | 46 |

- 500 并发下 QPS 与 100 并发基本持平，说明服务器吞吐能力稳定
- 延迟上升主要受限于线程池竞争与文件 I/O
